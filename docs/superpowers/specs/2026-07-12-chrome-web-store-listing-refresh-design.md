# Chrome Web Store 商店页刷新设计

日期：2026-07-12  
范围：Chrome Web Store 中文商店详情页、五张 1280×800 截图、仓库内商店文案源与提交检查清单。

## 1. 背景与目标

Chrome Web Store 当前公开页仍显示 `0.3.157`，概述只点名 B站、小红书、抖音和 YouTube；五张营销截图延续早期四平台与 B站中心叙事，其中设置截图也只展示四个平台。项目当前已经覆盖 B站、小红书、抖音、YouTube、X、知乎和 Reddit，并提供插件、桌面 Web、移动 Web、真实登录状态、收藏、稍后再看、画像、反馈和聊天等完整表面。

本次目标是让商店页在不夸大能力的前提下，第一眼传达三件事：

1. OpenBiliClaw 是本地优先、私有、开源的七平台内容发现 Agent；
2. 插件需要本地 OpenBiliClaw 后端，数据默认保存在用户本机；
3. 产品不是单纯采集器，而是覆盖跨平台推荐、画像、反馈、聊天和三端体验。

## 2. 视觉方向

采用「品牌首图 + 真实界面」混合方案：前两张集中表达定位和使用形态，后三张以最新版真实 UI 为主体。全部素材使用脱敏演示数据，不出现真实用户名、Cookie、API Key、后端密钥、浏览历史、私人画像或未公开账号信息。

截图统一规则：

- 尺寸固定为 1280×800，PNG 或高质量 JPEG；
- 延续现有浅色、粉蓝渐变、深蓝正文的品牌视觉；
- 标题不超过两行，每张只表达一个核心卖点；
- 真实 UI 必须来自当前 `main` 构建，不手绘不存在的控件；
- 平台名称统一为 B站 / 小红书 / 抖音 / YouTube / X / 知乎 / Reddit；
- 不展示后台自动访问平台、云端托管或无需后端等不实暗示。

## 3. 五张截图

### 3.1 本地私有的七平台内容 Agent

品牌首图。左侧一句话定位，右侧组合最新版插件推荐界面与七平台标识。文案强调「先理解你，再跨平台找内容」和「数据默认留在本机」，不再使用「让 B 站推荐像私人片单」的单平台叙事。

### 3.2 插件、桌面、手机一套体验

展示插件 side panel、桌面 Web 与移动 Web 的真实界面组合。说明三端共用同一本地后端、收藏、稍后再看和画像状态；二维码仅作产品示意，不包含真实局域网地址。

### 3.3 跨平台推荐，不只是采集

以最新版桌面或插件推荐流为主体，展示至少三个不同平台的来源徽标、推荐理由、收藏、稍后再看、去看看和聊天入口。示例内容使用公开、无敏感信息的演示数据。

### 3.4 画像会学习，也能纠正

展示最新版画像页、兴趣探针与避雷反馈。文案说明画像来自用户授权范围内的行为和反馈，并允许用户通过明确操作与聊天持续修正；不把推断描述为医学、心理诊断或确定事实。

### 3.5 登录状态诚实，数据留在本机

展示最新版平台设置页：七平台状态、`ready / unverified / login_required` 等真实语义，以及本地后端说明。强调小红书 `xsec_token` 只是内容令牌、抖音 Cookie 需要实际任务验证、Reddit 状态不会为了展示而访问平台。敏感字段全部隐藏或替换成明显的演示占位。

## 4. 商店文案结构

### 4.1 Short Description

短描述应在 Chrome Web Store 限制内，直接覆盖「本地优先」「七平台」「推荐与画像」三个关键词，不堆叠安装步骤或技术名词。

### 4.2 Detailed Description

中文主文案按以下顺序组织：

1. 两句定位：本地优先、七平台、需要本地后端；
2. 核心能力：跨平台信号、推荐理由、画像、反馈、收藏、稍后再看、聊天；
3. 支持平台：完整列出七个平台；
4. 四步安装：安装插件、下载并启动桌面后端、登录所选平台、打开侧边栏初始化；
5. 隐私边界：默认本地 SQLite、自带 LLM / embedding 配置、无开发者云端；
6. 项目主页、GitHub、Release、隐私政策和英文 README 链接。

文案不再把 B站写成必须登录的唯一入口；用户可以选择任一已经支持画像信号的登录平台。英语用户统一引导到 `README_EN.md`，避免详细说明中中英逐句混排。

## 5. 素材生成与可复现性

仓库新增商店截图素材目录和生成说明。真实 UI 截图优先使用现有本地演示 / 测试数据路径；如果现有 fixture 无法覆盖七平台来源徽标和最新状态枚举，则增加只服务截图的确定性演示数据生成步骤，不写入用户真实数据库。

最终提交到商店的五张图同时保存在仓库文档素材目录，文件名带稳定序号和语义，例如 `01-local-seven-platforms.png`。商店文案源继续维护在 `docs/chrome-webstore-listing.md`，并列出每张截图文件、展示目的和 dashboard 上传顺序。

## 6. 提交与审核

Chrome Web Store listing metadata 和截图只能在 Developer Dashboard 修改；现有 API v2 workflow 继续负责插件包上传与发布，不能冒充详情页更新。若 `0.3.163` 仍在审核，先撤回当前提交，更新详情和截图，再重新提交同一个已上传包审核；不得创建声称包含新插件代码的虚假版本。

提交前检查：

- 五张图均为 1280×800 且无敏感信息；
- Short / Detailed Description 与仓库文案源逐字一致；
- Website、Support、Privacy Policy URL 有效；
- 公开页不再只列四个平台；
- 截图和文案不声称后台状态页会访问平台；
- Developer Dashboard 显示新素材已经进入待审提交。

## 7. 验证

本地验证包括图片尺寸、文件格式、文案必备链接、七平台名称与敏感词扫描。视觉验证检查 1280×800 完整画布、标题层级、真实 UI 可读性和五张图顺序。提交后记录 Chrome Web Store 审核状态；只有 Dashboard / API 返回新 listing 已提交审核，才可报告「已提交」，公开页仍显示旧内容时必须说明审核尚未完成。

## 8. 非目标

- 不修改插件运行逻辑、权限或数据采集范围；
- 不新增云端服务或遥测；
- 不提交 Firefox AMO 商店详情；
- 不使用用户真实账号数据制作营销素材；
- 不因为 listing 更新而复用或移动已经发布的 `0.3.163` tag。
