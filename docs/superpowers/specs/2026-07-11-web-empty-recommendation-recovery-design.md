# Web 空推荐恢复设计

**日期：** 2026-07-11  
**范围：** 移动 Web `/m`、桌面 Web `/web`  
**不在范围：** 浏览器扩展推荐页、推荐接口语义、候选池计数口径

## 问题

移动 Web 对 `/api/runtime-status` 和 `/api/recommendations` 分别设置 5 秒与
12 秒超时，桌面 Web 的普通读取默认设置 15 秒超时。后端繁忙时，这些接口
可能晚于客户端预算返回。当前 Web 会把失败结果折叠成 `null` 或空数组，随后
展示成没有推荐或没有库存；浏览器扩展没有同样的短超时，因此最终仍能展示
同一后端里的库存和推荐。

移动 Web 首次请求结束后会把推荐视图标记为已加载，切换 tab 不再请求。
三端此前又刻意把 `refresh.pool_updated` 定义为“库存状态事件”而不是“推荐列表
替换事件”，避免覆盖用户滚动追加的卡片。两项行为组合后，Web 的首次瞬时失败
可能长期停留在空状态。

## 目标

1. 请求失败不能被解释成真实零库存或真实空推荐。
2. 移动 Web 与桌面 Web 的推荐列表或运行时库存状态首次读取失败后可以分别自动
   恢复。
3. 自动恢复只能发生在当前推荐列表为空时，绝不覆盖已有或追加的卡片。
4. 同一时间只允许一个恢复请求，连续失败使用有上限的退避，成功后停止。
5. 后端明确成功返回空列表时，继续展示真实空库存提示，不进入无限重试。

## 方案

采用“空列表条件恢复”，不单纯扩大超时，也不把后端性能优化作为本次修复的
前置条件。

### 移动 Web

- `loadData()` 保留推荐请求的成功/失败区别，不再用 `.catch(() => [])` 抹平错误。
- 新增视图内恢复状态：推荐读取和运行时状态读取各自是否失败、单飞中的
  Promise/标记、重试次数和计时器。
- 首次失败时保留空列表，渲染“加载失败，正在重试”语义，并调度有上限的退避
  重试。
- 以下信号只有在“当前列表为空且最近一次推荐读取失败”时触发或提前执行推荐
  恢复：
  `refresh.pool_updated`、runtime stream 重新连接后重新进入推荐页、用户切回推荐
  tab。
- 成功返回任意数组（包括空数组）都结束失败恢复状态。成功非空时渲染卡片；
  成功空数组时走原有真实空库存文案。
- 运行时状态读取失败后独立重试；任一含库存字段的 `refresh.pool_updated` 事件可
  直接恢复库存状态并结束该轮状态重试，不要求同时重新拉推荐列表。

### 桌面 Web

- 初始 hydration 单独记录 `/api/recommendations` 与 `/api/runtime-status` 是否失败，
  避免 `requestJson()` 的 `null` 被当成空成功。
- 增加与移动端同语义的空列表恢复调度器，并复用已有 hydration 防并发模式。
- `refresh.pool_updated` 仍不做全量 hydration；只有当前 `state.videos` 为空且最近
  一次推荐读取失败时，才单独重试推荐列表。
- runtime status 失败独立重试；收到含库存字段的实时事件后立即结束状态重试。
- `config_reloaded`、初始化完成等现有全量 hydration 行为保持不变。

## 退避与并发

- 单飞：每类恢复请求执行期间收到更多触发信号，只记录一次 pending，不并发同类
  请求；推荐和运行时状态可以独立恢复。
- 退避：连续失败按 1 秒、2 秒、4 秒、8 秒增长，最大 8 秒。
- 自动重试次数上限为 4；达到上限后保留可手动重试的错误提示。新的 WebSocket
  重连、`refresh.pool_updated` 或重新进入推荐页可以开启新一轮恢复。
- 对应资源成功后清空该资源的 timer、次数、失败标记和 pending 状态。

## UI 与错误语义

- `loading`：继续使用现有 spinner/skeleton。
- `failed`：显示“推荐加载失败，正在重试”，达到本轮上限后显示“推荐加载失败，
  点此重试”。
- `empty-success`：使用现有“当前列表里的推荐都已处理/候选池正在补货”等文案。
- 库存数字只有拿到 runtime status 或含库存字段的实时事件后才更新；未知状态不
  渲染成零库存。

## 测试

先写失败测试，再实现：

1. 移动 Web 首次推荐请求失败后不会把失败记成成功空列表，并会调度恢复。
2. 移动 Web 首次 runtime status 请求失败后独立重试；含库存字段的事件会结束状态
   重试。
3. 移动 Web 在列表非空时忽略 `refresh.pool_updated` 的列表刷新。
4. 移动 Web 成功返回空数组后停止推荐失败恢复。
5. 桌面 Web 初次读取推荐或 runtime status 返回 `null` 时分别记录失败并调度恢复。
6. 桌面 Web 列表非空时库存事件不会替换列表。
7. 两端两类恢复调度器满足单飞、退避上限和成功清理。

## 文档影响

- 更新 `docs/modules/recommendation.md`，说明 Web 首次读取失败与真实空库存分离，
  且库存事件只在空失败态触发条件恢复。
- 更新 `docs/changelog.md` 当前版本条目。
- 不改变接口、模块边界、配置、CLI 或跨模块数据流，因此不需要更新架构图、
  配置文档或 CLI 文档。

## 验收标准

- 在推荐或运行状态接口超过 Web 超时预算的情况下，Web 明确显示加载失败并自动
  恢复，而扩展/Web 不再出现“同一库存、不同真假空状态”。
- 恢复成功后无需刷新整个页面即可显示推荐和真实库存。
- 用户已经看到或追加的卡片不会被任何库存事件覆盖。
- 针对性测试、Web 静态/Node 测试、Python 测试以及 Ruff 检查通过。
